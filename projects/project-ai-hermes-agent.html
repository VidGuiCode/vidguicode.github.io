Hermes AI Agent

From May 2026 to today: my personal AI agent running on its own VPS, connected to my Matrix server, ntfy notifications, Groq transcription, Piper TTS, homelab endpoints, email, calendar, transport lookups, git server, voice notes, image generation, and a markdown knowledge base.

Last updated: 15/06/2026

The Idea

Hermes is not just a chatbot I open when I need help. It is a personal AI workspace that runs continuously on its own VPS while the communication layer lives inside my infrastructure. I set up a local Synapse server so Hermes can talk to me mainly through Matrix, while ntfy handles push notifications for things that should not wait.

Why I Built It

I wanted an assistant that could keep context across days, work with my own services, and handle small recurring tasks without me opening five different tools. The important part was not making a flashy bot. It was connecting AI to real daily workflows: voice notes, research, public transport, reminders, calendar events, email checks, homelab status, coding tasks, and personal knowledge.

How I Use It

Most interaction happens through Matrix. I can write to Hermes, send voice notes, receive Groq-powered transcriptions back, ask for research on topics that matter to me, and get the final report delivered in the same place. When I am too lazy to read, Hermes can answer back with Piper TTS so the interaction feels more natural. When something is time-sensitive, Hermes can also push it through my self-hosted ntfy setup.

Core Workflows

Matrix & Platform Communication

Synapse & ntfy integration

Hermes is controlled through Matrix, maintaining long-term session context. It processes links, transcribes voice notes with Groq STT, responds with Piper TTS, and alerts via ntfy. Subagent delegation runs isolated workers for heavy lifting.

What Makes It Interesting

  • Voice notes make the agent usable when typing is annoying: Groq handles free transcription, and Piper can speak answers back when I do not want to read.
  • The transport lookup is not a generic API demo. It is connected to routes I actually use and can be extended with more predefined stations.
  • The knowledge base works like a markdown/Obsidian-style wiki stored through my git server, so Hermes can retrieve durable context instead of relying only on chat memory.
  • Hermes can also generate images, which made model routing more important because text, vision, and image tasks benefit from different providers.
  • The agent runs in Docker with persistent folders and recovery tooling, so losing a session or breaking a tool does not mean losing everything.

Provider and Cost Lessons

This project taught me a lot about AI providers, subscriptions, pay-as-you-go pricing, and session hygiene. One expensive Alibaba PAYG week cost me 107 EUR because I did not understand the platform well enough and let context usage get out of control. That made the lesson very concrete: compacting, starting clean sessions, and choosing the right model for the task are not small optimizations. They decide whether an agent is affordable.

Because I do not have the hardware to run strong local models, I depend on external providers. Hermes helped me test many paid and free models, and the auxiliary model setup showed me a better pattern: use a cheaper text model as the main worker, route vision to a vision-focused model, and only use stronger models when the task actually needs them.

Isolation and Recovery

Running Hermes in Docker instead of bare metal created extra problems at first, but it also forced a cleaner architecture. I had to think about persistent folders, what the agent is allowed to touch, how it can watch its own state, and how it can recover if a tool or session breaks. That made the system more reliable than a quick one-off install.

Lessons Learned

  • Memory is the differentiator: A chatbot without memory is a search engine with extra steps.
  • Cost control beats model quality: Don't burn Claude on "is Docker up". Cheap models handle routine checks fine.
  • Skills beat prompts: A one-shot prompt is disposable. A skill that gets patched when it breaks compounds over time.
  • Self-hosting teaches the stack: No managed service to blame — you own the infrastructure, networking, and secrets.
  • Subagents prevent context flooding: Isolating heavy tasks inside worker subagents keeps the main conversation clean.
  • Cron jobs stay boring: Keep automation pipelines simple — collect data, summarize, and deliver. No complex state machines.
  • Platform routing is invaluable: Sending to Matrix, ntfy, or local files without rewriting the job makes workflows modular.

Next Steps

Next I want to expand the homelab integration carefully, add more useful transport destinations, improve the skills that Hermes uses for its own tooling, and continue tuning provider/model routing so the agent stays useful without becoming expensive.
