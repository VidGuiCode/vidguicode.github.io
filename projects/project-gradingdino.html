We worked as a team of four with clearly divided responsibilities:

  • My Role (Lead Developer): I handled most of the coding â€” Django models, views, templates, and Docker configuration. I also collaborated on database design and helped structure the documentation.

What We Built

The system provides role-based access for different users:

  • Student Portal: Students log in to view their grades across subjects, track progress by semester, and write personal notes.
  • Teacher Dashboard: Teachers create classrooms, enroll students, assign grades with comments, and write private observations.
  • Admin Panel: Django's built-in admin for managing users, subjects, grading scales (20-point, 60-point, letter grades).
  • Docker Deployment: Single docker compose up command starts the entire system â€” PostgreSQL + Django.

Why Django (And What We Learned)

We researched Django, Flask, and FastAPI before choosing. Django won because of its "batteries included" approach â€” built-in admin panel, authentication, forms, and ORM meant we could focus on features instead of boilerplate.

However, "batteries included" doesn't mean "no learning required." Django has a steep learning curve with its own conventions, project structure, and way of doing things. With our tight deadline, we didn't have enough time to properly understand the framework â€” we were building while learning, which slowed us down.

The honest truth: The project was an MVP that worked but wasn't finished. It taught us that choosing a framework you don't know adds hidden time costs â€” the "batteries" are only useful if you know how to use them.

Lessons Learned

  • Communication is everything â€” We divided work to move faster, but this created silos. Misunderstandings cost us time we could have spent on features. In the future, we'll treat communication with the same priority as coding.
  • Framework selection matters â€” Django was great for rapid prototyping, but its conventions felt limiting when we needed flexibility. Research upfront saves pain later.
  • Planning isn't optional â€” Writing personas and user stories before coding felt slow, but it gave us clarity. We knew exactly what to build and could prioritize effectively.
  • Learn the tool before the deadline â€” Django's "batteries included" only help if you understand how to use them. We spent time fighting the framework instead of building features.
  • MVP is okay â€” The project wasn't perfect, but it worked. Shipping something functional taught us more than endless planning ever could.
