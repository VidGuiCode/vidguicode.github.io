The Project

GradingDino was a BTS Cloud Computing group project focused on project management methodology. A team of four worked together to build a grade management web application where teachers can create classrooms, add students, and manage grades — while students can log in to view their progress and leave personal notes.

Team Structure & My Role

We worked as a team of four with clearly divided responsibilities:

  • My Role (Lead Developer): I handled most of the coding — Django models, views, templates, and Docker configuration. I also collaborated on database design and helped structure the documentation.
  • Colleague (Database & Docs): Worked with me on database schema design and took ownership of technical documentation.
  • Two Colleagues (Planning): Created personas, wrote all 12+ user stories with acceptance criteria, performed MoSCoW prioritization, and managed the MS Planner board.

Under the Hood

Five containerized layers, all booted together by a single Compose file:

Presentation

Pico.cssSemantic HTML5Server-side renderingDjango Templates
  • Pico.css: a ~10KB single-file framework that styles semantic HTML5 automatically — no build step, no learning curve, just write HTML.
  • Server-side rendering: Django renders every page and the browser posts back over a standard HTTP cycle, keeping all validation and CSRF protection on the server.

Application

Django 5.0Python 3.11-slimDjango ORMAdminFormsSessions
  • Batteries included: Django 5.0 on Python 3.11-slim shipped auth, forms, an ORM and an admin out of the box, so a small team could build features instead of plumbing.
  • Built-in admin: the Django admin manages users, subjects and grading scales (20-point, 60-point and letter grades) with no extra code.
  • ORM models: classrooms, enrolments, grades and notes are plain Python models; migrations are generated and applied automatically.

Data

PostgreSQL 16ACIDConcurrent writesNamed volume
  • PostgreSQL 16: an ACID-compliant relational store that survives container restarts through a named Docker volume.
  • Concurrent writes: several teachers can grade at the same time without locking each other out — and the same engine scales to a production deployment.

Security

Argon2CSRFXSS escapingSQLi-safe ORMSession auth
  • Argon2 hashing: winner of the Password Hashing Competition (2015), created at the University of Luxembourg — memory-hard and resistant to GPU/ASIC cracking. A proud local choice.
  • Secure by default: Django ships CSRF tokens, template auto-escaping against XSS, and a parameterised ORM that blocks SQL injection.
  • Session auth + roles: signed session cookies plus role checks keep students, teachers and admins to their own views.

Orchestration

DockerDocker ComposeHealth checksAuto-migrationsPersistent volumes
  • One command: docker compose up builds and starts the web and database containers together — identical on any machine.
  • Health-gated startup: the app waits for PostgreSQL to pass a health check, then runs migrations and seeds the admin user before serving traffic.
  • Persistent volumes + live reload: data survives restarts via named volumes, and mounted code hot-reloads during development for fast iteration.

Why Django — and what it taught us

01

Before a line of code, we did the groundwork: personas, 12+ user stories with acceptance criteria, MoSCoW prioritisation, and an MS Planner board. Planning first gave us a clear picture of exactly what to build.

02

We compared Django, Flask and FastAPI. Django won on its "batteries included" philosophy — a built-in admin, auth, forms and ORM meant a small team could move fast without wiring everything from scratch.

03

The catch: Django wants things done its way. On a tight deadline we hadn't fully learned its conventions, so we spent time fighting the framework instead of building features. We still shipped a working MVP — and learned the "batteries" only pay off once you know how to use them.

Lessons Learned

Communication beats speed

Splitting work to move faster created silos. Next time, talking to each other gets the same priority as coding.

Framework choice matters

Django was great for prototyping but felt rigid when we needed flexibility. Research up front saves pain later.

Planning isn't optional

Personas and user stories felt slow at first, but they told us exactly what to build and what to skip.

Learn the tool first

Django's batteries only help once you know them. Under deadline pressure we fought the framework instead of using it.

A working MVP wins

It wasn't perfect, but it ran. Shipping something functional taught us more than endless planning could.
